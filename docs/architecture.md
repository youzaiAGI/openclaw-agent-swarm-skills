# Architecture

OpenClaw Agent Swarm is designed to provide a robust and isolated execution layer for AI coding agents.

## 1. Core Principles

- **Task Isolation**: Each task runs in its own **Git Worktree** and **Branch**.
- **Asynchronous Execution**: Tasks are detached from the coordinator and run in **Tmux sessions**.
- **State Management**: Task metadata, logs, and state transitions are tracked via local JSON files.

## 2. Execution Layers

### Coordinator Layer
The coordinator (usually an external LLM agent or runtime) issues commands to the `swarm.js` entry point. It polls for changes via `check --changes-only`.

### Swarm Skill Layer (`swarm.js`)
This layer handles the lifecycle of a task:
1. **Validation**: Checks for tools and git repository status.
2. **Setup**: Creates a Git worktree and a new branch (`swarm/<task_id>`).
3. **Execution**: Generates a prompt and starts an Agent CLI within a Tmux session.
4. **Monitoring**: Polls Tmux and exit files to update task state.

### Task Layer (Agent CLI)
The actual coding agent (Codex, Claude, etc.) operates inside the worktree. It reads the prompt file generated by the swarm and performs the task.

## 3. Storage Structure

By default, all persistent state is stored in `~/.agents/agent-swarm/`:

- `tasks/`: Individual JSON files containing metadata for each task.
- `tasks/history/`: Archived tasks organized by date.
- `worktree/`: Managed Git worktrees for active tasks.
- `logs/`: Combined stdout/stderr logs from the Agent CLI.
- `prompts/`: Instructions passed to the Agent CLI.
- `agent-swarm-last-check.json`: Tracking file for incremental status updates.

## 4. State Machine

Tasks transition through the following states:

- **running**: The task is actively executing in a Tmux session.
- **pending**: The interactive session is waiting for user/coordinator input (no activity).
- **success**: The task completed with an exit code of 0.
- **failed**: The task exited with a non-zero code or encountered an error.
- **stopped**: The task was manually cancelled or the Tmux session was killed.

## 5. Security & Isolation

- **File System**: Agents are instructed to operate only within their assigned worktree.
- **Session Isolation**: Each agent runs in a unique Tmux session, preventing cross-task interference.
- **Concurrency**: File locks (`flock`) ensure that concurrent status checks don't corrupt the state files.
